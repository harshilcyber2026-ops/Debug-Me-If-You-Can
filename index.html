⚙ Reverse Engineering · Challenge #07

Debug Me If You Can

A Linux binary with three anti-debugging layers: ptrace detection, rdtsc timing checks, and CRC32 code integrity. Bypass all three to reveal the flag.

Description

You've been handed a suspicious ELF binary. It claims to hold a flag, but it actively fights back against anyone trying to analyze it.

The binary implements three independent anti-debug layers. All three must pass simultaneously for the flag to be revealed. A single failure — any running debugger, any timing anomaly, any patched byte — and you get nothing.

Your goal: understand each protection mechanism and neutralize it without triggering the others.

Protection Layers

01
Layer One
ptrace Detection
Calls PTRACE_TRACEME on itself. If a debugger is attached, this fails with EPERM (-1).
02
Layer Two
Timing Check
Uses RDTSC to measure a tight loop. Under a debugger, the delta vastly exceeds the 500k cycle threshold.
03
Layer Three
CRC32 Integrity
Computes CRC32 of its own .text at runtime. Any patched byte — a breakpoint, a NOP — alters the checksum.

Bypass Strategy

Method A — LD_PRELOAD (recommended): Override ptrace() with a stub that returns 0. No binary bytes change → CRC stays valid. Since you're not in a debugger, timing is also fast.

Method B — GDB + scripting: Hook the ptrace syscall return, override the rdtsc delta, and force the CRC comparison result. Complex but teaches all three layers deeply.

Method C — Binary patch: NOP out the three conditional jumps. Warning: NOP bytes alter CRC — you must also update the stored expected CRC value at g_expected_crc.

⚑ The flag bytes are XOR-obfuscated — strings/xxd won't reveal it. You must run the binary with all checks passing.
